## 2. 현재 강의평가 항목과 문제점 (감사 결과)

### 2-1. 학생 피드백 폼 (`src/app/feedback/[courseId]/feedback-form.tsx`)

| # | 화면 문항 | DB 필드 | 입력 형태 | 비고 |
|---|-----------|---------|-----------|------|
| 1 | 이번 수업의 진행 속도 | `speed` (String) | 3점 범주 `fast`/`moderate`/`slow` | 필수 |
| 2 | 강의 내용 이해도 | `comprehension` (String) | 3점 범주 `high`/`medium`/`low` | 필수 |
| 3 | 질문·소통 만족도 | `communication` (Int) | 1~5 | 필수 |
| 4 | 수업 흥미도 | `interest` (Int?) | 1~5 | 항상 표시(현재) |
| 5 | 과제 난이도·분량 적절성 | `assignment` (Int?) | 1~5 | `hasAssignment`일 때만 |
| 6 | 실습/예시 충분도 | `practice` (Int?) | 1~5 | `hasPractice`일 때만 |
| 7 | 자유 의견 | `comment` (String?) | 자유 서술(≤500자) | 선택 |

제출: `submitFeedback`(1회용 토큰) / `submitStudentFeedback`(다회용) → 검증(`VALID_SPEED`/`VALID_COMPREHENSION`, 1~5 범위) → 저장 → `comment`는 백그라운드 AI 분류(`backgroundClassify`).

### 2-2. 핵심 문제점

1. **척도 불일치 (가장 큰 문제).** 1~2번은 3점 범주형, 3~6번은 1~5 Likert.
   - 레이더 정규화가 축마다 다름: 속도=`%moderate`, 이해도=`%high`, 나머지=`평균/5×100`.
   - 결과적으로 레이더의 한 축은 "적정이라고 답한 비율", 다른 축은 "평균 만족도" → **축끼리 직접 비교 불가**, 교수가 "어디가 약한지" 읽기 어려움.
2. **속도의 방향 정보 손실.** `%moderate`만 지표화하면 "너무 빠름"과 "너무 느림"이 똑같이 "적정 아님"으로 묶여 **개선 방향(빠르게/느리게)을 알 수 없음**.
3. **흥미도(interest)의 평가성 톤.** "수업 흥미도"는 교수에게 인기투표처럼 읽힐 수 있어 V3 제품 방향("교수가 평가받는 느낌 지양")과 충돌. 참여/도움 관점으로 재프레이밍 여지.
4. **널러블 축의 들쭉날쭉함.** `interest`/`assignment`/`practice`가 nullable + 조건부라 강의마다 레이더 축 개수(4~6)가 달라지고, 코스 간 비교(벤치마크)가 불안정.
5. **강의자료 분석에 무관 지표 혼입.** `analyze-material.ts`가 소통·흥미·과제·실습까지 전부 프롬프트에 투입 → 자료 품질과 무관한 신호가 자료 분석을 흐림(사용자 지적 사항).
6. **자유 의견 단일 필드.** 좋았던 점/어려웠던 점/제안이 한 덩어리 → 어조 분류(방금 추가한 긍정/중립/부정)로 일부 보완되나, 구조화 데이터로서의 가치는 낮음.

## 3. 새 평가 항목 후보

핵심 원칙: **(a) 모든 만족도 축을 1~5로 통일**(레이더 비교 가능), **(b) 방향이 필요한 "속도"만 별도 처리**, **(c) 자료 신호를 명시적으로 추가**해 강의자료 분석을 정당화, **(d) 평가성 톤 완화**.

### 3-A. 권장안 (Recommended) — "코어 4축(1~5) + 속도(방향) + 조건부 2축"

| 지표 | 척도 | 방향 | 비고 |
|------|------|------|------|
| **내용 이해** | 1~5 | 높을수록 좋음 | 기존 `comprehension`(범주) 대체 |
| **자료 도움** | 1~5 | 높을수록 좋음 | **신규**. 강의자료 분석의 정당한 입력 신호 |
| **소통·질문 편의** | 1~5 | 높을수록 좋음 | 기존 `communication` 유지/리네이밍 |
| **참여 의욕** | 1~5 | 높을수록 좋음 | 기존 `interest`(흥미도) 재프레이밍(덜 평가적) |
| 수업 속도 | 3점(느림/적당/빠름) | 방향 지표 | 만족도 레이더와 분리, 별도 "속도 보정" 게이지로 표시 |
| 과제 적절성 *(조건부)* | 1~5 | 높을수록 좋음 | `hasAssignment`일 때만 |
| 실습/예시 도움 *(조건부)* | 1~5 | 높을수록 좋음 | `hasPractice`일 때만 |

- 레이더 = 코어 4축(+조건부 최대 2축) **모두 1~5 평균** → 축 의미 통일.
- 속도는 레이더에서 빼고 **별도 분포 막대/게이지**(느림 N% · 적당 N% · 빠름 N%)로 → 방향 정보 보존, 개선 액션 명확.
- "자료 도움"이 생기면 강의자료 분석은 **이해+자료 도움(+속도)** 만 입력 → 무관 지표 혼입 해소.

### 3-B. 대안 (Low-cost) — "문구만 정리, 척도 유지"
- DB/마이그레이션/차트 정규화 거의 그대로, 화면 문구·헬프텍스트만 개선.
- 장점: 위험·공수 최소. 단점: 척도 불일치(2-2-1)와 자료 혼입(2-2-5)은 **미해결**.
- 마감 임박 시 fallback.

### 3-C. 대안 (Minimal-axis) — "코어 3축만"
- 내용 이해 / 자료 도움 / 소통 3개 1~5 + 속도(방향) + 자유 의견. 흥미·과제·실습 제거.
- 장점: 학생 부담 최소, 해석 단순. 단점: 과제/실습 강의의 변별력 상실.

> **권장: 3-A.** 이하 4~8장은 3-A 기준으로 작성하되, 3-B/3-C 선택 시 차이를 각 장에 표기.

## 4. 학생 화면 문항 문구 (3-A 기준 초안)

모든 1~5 문항은 "그렇다" 방향 통일 (5 = 매우 그렇다).

1. **내용 이해** — "오늘 수업 내용을 잘 이해할 수 있었다." (1 전혀 아니다 ~ 5 매우 그렇다)
2. **자료 도움** — "강의 자료(슬라이드·예제 등)가 이해에 도움이 되었다."
3. **소통·질문 편의** — "질문하거나 의견을 말하기 편했다."
4. **참여 의욕** — "이 수업에 더 집중/참여하고 싶었다." (흥미도의 재프레이밍)
5. **수업 속도** — "수업 진행 속도는 어땠나요?" → 느림 / 적당 / 빠름 (단일 선택)
6. *(과제 강의)* **과제 적절성** — "과제의 난이도·분량이 적절했다."
7. *(실습 강의)* **실습/예시 도움** — "실습·예시가 이해에 도움이 되었다."
8. **자유 의견(선택)** — §7 결정에 따름.

문구 원칙: 한 문장·평서문·"오늘/이 수업" 시점 고정, 부정문 지양, 5점 라벨 양 끝만 명시.

## 5. 교수 대시보드 지표 매핑 (3-A 기준)

| 학생 항목 | 대시보드 표현 | 차트 |
|-----------|---------------|------|
| 내용 이해(avg) | KPI "이해" + 레이더 축 | 레이더/막대(평균/5) |
| 자료 도움(avg) | KPI "자료" + 레이더 축 | 레이더/막대 |
| 소통(avg) | KPI "소통" + 레이더 축 | 레이더/막대 |
| 참여(avg) | 레이더 축 | 레이더/막대 |
| 속도(분포) | **별도 "속도 보정" 카드** (느림/적당/빠름 %) | 가로 분포 막대(레이더 제외) |
| 과제/실습(avg, 조건부) | 레이더 축(해당 강의만) | 레이더/막대 |
| 자유 의견 | 대표 학생 의견 + 어조 그룹 | 기존 |

- KPI 4칸 후보: **총 응답 / 내용 이해 / 자료 도움 / 소통** (또는 참여). 속도는 KPI가 아니라 보정 카드.
- 추세 차트: 1~5 축(이해/자료/소통/참여) + 속도 적정% 별도 라인 → **모두 0~100 정규화(×20)로 한 그래프** 가능.
- 벤치마크/개선사례의 `communicationAvg`/`comprehensionHighRatio`/`speedModerateRatio` 키 → 새 스키마에 맞춰 재정의 필요(§7).

## 7. DB / Prisma / 마이그레이션 영향

### 7-1. 스키마 변경(3-A)
- `comprehension: String` → `comprehension: Int`(1~5) **혹은 신규 `comprehensionScore Int` 추가 후 구필드 제거** (마이그레이션 안전성).
- `materialHelp Int?` **신규** (자료 도움). 코어 축이므로 사실상 필수지만, 구데이터 호환 위해 nullable로 추가 후 점진 필수화 검토.
- `interest` 유지(라벨만 "참여"로) 또는 `engagement`로 리네이밍.
- `speed: String` 유지(방향 지표라 범주형이 맞음).
- `assignment`/`practice` 유지.

### 7-2. 마이그레이션 전략 (택1, §승인 필요)
- **(가) 클린 리셋** — 데모 앱이므로 기존 Feedback 폐기 후 새 스키마로 재시드. 가장 단순/안전. **권장**(공모전 데모 특성).
- **(나) 매핑 보존** — 기존 `comprehension` high/medium/low → 5/3/1 변환, `materialHelp`는 NULL 백필. 실데이터 보존 필요 시.

> Prisma 7 + SQLite + driver adapter 환경. `String→Int` 타입 변경은 SQLite에서 컬럼 재생성이 필요하므로 **신규 컬럼 추가 → 백필 → 구컬럼 제거**의 2단계 마이그레이션이 안전.

### 7-3. 영향 받는 검증/타입
- `feedback.ts`의 `VALID_SPEED`/`VALID_COMPREHENSION` → comprehension이 Int가 되면 범위검증으로 교체.
- `feedback-stats.ts` `FeedbackCountInput`/`computeFeedbackCounts` 시그니처 변경(범주 카운트 → 평균 합산).
